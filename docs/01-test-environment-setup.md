# 実店舗テスト環境 構築ガイド

## 目的

NIGHTOSを実店舗で使ってもらうために必要な環境を整える。
現在の「モックデータ + localStorage」から「実データ + 永続化」への移行。

---

## 現状と課題

### 今できていること
- ✅ 全画面が動作する MVP（キャスト / 店舗 / 来店客）
- ✅ Vercel にデプロイ済み（公開 URL あり）
- ✅ Claude Haiku 4.5 によるさくらママ AI
- ✅ PWA 対応（ホーム画面に追加可能）

### 実店舗テストに必要だが未対応のこと
- ❌ データの永続化（現状はサーバー再起動で消える）
- ❌ ユーザー認証（現状は localStorage のロール切替のみ）
- ❌ 店舗ごとのデータ分離（現状は全員が全データを見れる）
- ❌ 複数キャストの同時利用（現状は「あかり」固定）

---

## 段階的な移行プラン

### フェーズ1: 最小限の永続化（1〜2日）

**目標**: サーバー再起動してもデータが消えない状態にする

**やること**:
1. Supabase Cloud プロジェクトを作成（無料）
2. SQL Editor で `002_nightos_schema.sql` を実行
3. SQL Editor で `seed_nightos.sql` を実行（デモデータ投入）
4. Vercel に `NEXT_PUBLIC_SUPABASE_URL` と `NEXT_PUBLIC_SUPABASE_ANON_KEY` を設定
5. Redeploy

**結果**: データが Supabase のPostgreSQLに永続化される。mock fallback が自動的に DB モードに切り替わる。

**リスク**: 現時点では Supabase 実クエリが未テストのため、初回接続時にエラーが出る可能性あり。その場合は mock にフォールバックする設計なので致命的ではない。

### フェーズ2: 簡易認証（2〜3日）

**目標**: 「誰がログインしているか」を識別できるようにする

**選択肢**:

#### A. PIN コード認証（最も簡単）
- 店舗ごとに4桁のPINを発行
- キャストごとに個別のPIN
- ログイン画面でPIN入力 → 該当ロール + ユーザーが確定
- localStorage でセッション管理
- Supabase Auth は不要

```
店舗PIN: 1234 → 店舗スタッフとしてログイン
あかりPIN: 5678 → キャスト「あかり」としてログイン
みさきPIN: 9012 → キャスト「みさき」としてログイン
```

**メリット**: 実装が簡単、キャストにとって入力が楽
**デメリット**: セキュリティは弱い（テスト段階ではOK）

#### B. Supabase Auth（本格的）
- メールアドレス + パスワードでログイン
- Supabase の RLS でデータアクセス制御
- ロール（store_staff / cast / customer）をユーザーメタデータに保存

**メリット**: セキュリティが高い、将来拡張しやすい
**デメリット**: 実装に時間がかかる、キャストにメアド登録させる必要あり

**推奨**: テスト段階は **A（PINコード）** で始める。本番移行時に B に切り替え。

### フェーズ3: 店舗のデータ分離（1日）

**目標**: テスト店舗のデータが他の店舗から見えないようにする

**やること**:
- `CURRENT_STORE_ID` を環境変数化（`NEXT_PUBLIC_STORE_ID`）
- 各クエリに store_id フィルタを追加（大部分は既に実装済み）
- テスト店舗ごとに別の Vercel プロジェクト or 環境変数で分離

### フェーズ4: 複数キャスト対応（1日）

**目標**: キャストごとに自分のデータだけ見える

**やること**:
- `CURRENT_CAST_ID` をログインユーザーから取得するように変更
- PIN 認証で cast_id が確定 → サーバー側で使用

## テスト店舗のオンボーディング手順

### 事前準備（営業担当が行う）

1. Supabase に店舗のデータを作成
   ```sql
   INSERT INTO nightos_stores (id, name) VALUES ('store_xxx', '店舗名');
   INSERT INTO nightos_casts (id, store_id, name) VALUES ('cast_xxx', 'store_xxx', 'キャスト名');
   ```

2. Vercel で環境変数を設定
   - `NEXT_PUBLIC_STORE_ID=store_xxx`

3. テスト用 PIN を発行して店舗に伝える

### 店舗スタッフの初日

1. スマホで NIGHTOS の URL を開く
2. 「ホーム画面に追加」でアプリ化
3. 店舗スタッフ PIN でログイン
4. 主要顧客を 5〜10人登録（1人2分）
5. ボトルキープ情報を登録

### キャストの初日

1. スマホで NIGHTOS の URL を開く
2. 「ホーム画面に追加」
3. キャスト PIN でログイン
4. ホーム画面で連絡リストを確認
5. さくらママに1件相談してみる
6. テンプレートで1件 LINE を送ってみる

## テスト期間の運用

### 毎日のルーティン

| 時間帯 | 誰が | 何をする |
|--------|------|----------|
| 開店前 | 店舗スタッフ | 予約客を来店登録の準備、ボトル在庫確認 |
| 来店時 | 店舗スタッフ | 来店登録（テーブル + 顧客 + 指名）|
| 営業中 | キャスト | さくらママに接客相談（必要なら）|
| 閉店後 | キャスト | 今日の顧客にお礼テンプレートを送る |
| 翌朝 | キャスト | ホーム画面で連絡リストを確認 → LINE |

### 週次レビュー

- 店舗ダッシュボードで連絡達成率を確認
- キャスト別の指名数推移を見る
- フィードバックをヒアリング（→ ドキュメント2参照）

## 技術的な注意点

### API コスト
- さくらママ（Claude Haiku 4.5）: 約 0.7円/回
- モーニングブリーフィング: 約 0.7円/日/キャスト
- ボトル提案・テンプレ生成・LINE 解析: 各 0.7円/回
- 月間見込み: キャスト5人×1日10回利用 = 約 1,050円/月

### Vercel の制限（Hobby プラン）
- 関数実行時間: 10秒まで（さくらママの応答は通常 2〜5秒なので OK）
- 帯域: 100GB/月（テスト段階では十分）
- 同時接続: 制限なし

### Supabase の制限（Free プラン）
- データベース: 500MB（テストには十分）
- ストレージ: 1GB（LINE スクショの保存に使う場合注意）
- API リクエスト: 無制限
- 同時接続: 制限あり（ピーク時 50 程度）
